## Overview

Mi Ruta Directa automatically detects the user's geographic location using the browser's **Geolocation API**. This feature helps users by centering the map on their current position, making it easier to find nearby addresses and plan routes from their location.

## useMyLocation Hook

The location detection logic is encapsulated in a custom React hook located at `src/hooks/useMyLocation.ts`.

### Hook Implementation

Here's the complete implementation from `useMyLocation.ts:4-29`:

```tsx src/hooks/useMyLocation.ts
import { useState, useEffect } from "react"
import { LatLng } from "leaflet"

export const useMyLocation = () => {

  const [position, setPosition] = useState<LatLng | null>(null);

  useEffect(() => {

    navigator.geolocation.getCurrentPosition(
      (pos) => {
        const { latitude, longitude } = pos.coords;
        setPosition(new LatLng(latitude, longitude));
      }, 
      () => {
        alert("No se pudo obtener tu ubicación");
      }, 
      {
        enableHighAccuracy: true,
        timeout: 5000,
        maximumAge: 0
      }
    );

  }, [])

  return {
    position
  }
}
```

### How It Works

<Steps>
  <Step title="Initialize State">
    Position is initialized as `null` using `useState<LatLng | null>(null)`
  </Step>
  
  <Step title="Request Location on Mount">
    The `useEffect` with empty dependency array `[]` runs once when the component mounts
  </Step>
  
  <Step title="Call Geolocation API">
    `navigator.geolocation.getCurrentPosition()` requests the user's location
  </Step>
  
  <Step title="Handle Success">
    On success, coordinates are converted to Leaflet's `LatLng` object and stored in state
  </Step>
  
  <Step title="Handle Failure">
    On failure, an alert notifies the user that location couldn't be obtained
  </Step>
  
  <Step title="Return Position">
    The hook returns the position object for use in components
  </Step>
</Steps>

## Geolocation API Configuration

The hook uses specific options when calling `getCurrentPosition()`:

```tsx src/hooks/useMyLocation.ts
{
  enableHighAccuracy: true,
  timeout: 5000,
  maximumAge: 0
}
```

### Configuration Options

| Option | Value | Description |
|--------|-------|-------------|
| `enableHighAccuracy` | `true` | Requests GPS-level accuracy (may take longer, uses more battery) |
| `timeout` | `5000` | Maximum time (5 seconds) to wait for location |
| `maximumAge` | `0` | Don't use cached location—always get fresh coordinates |

<Info>
  Setting `enableHighAccuracy: true` provides better precision but may drain mobile batteries faster. For applications where approximate location is sufficient, consider setting this to `false`.
</Info>

Implementation from `MapaControles.tsx:4-14`:

```tsx src/components/MapaControles.tsx
export const ObtenerUbicacionUsuario = ({ position }: { position: [number, number] | null }) => {
  const map = useMap();
  
  useEffect(() => {
    if (position) {
      map.flyTo(position, 15);
    }
  }, [position, map]);
  
  return null;
}
```

- Uses `map.flyTo()` for smooth animated transition
- Zooms to level 15 (street-level view)
- Triggers automatically when position becomes available

<Tip>
  The `flyTo()` method provides a better user experience than `setView()` by smoothly animating the pan and zoom transition instead of jumping instantly.
</Tip>

## Location Permissions

### How Browser Permissions Work

When `getCurrentPosition()` is called, the browser:

1. **Checks permission status** - Has the user already granted/denied location access?
2. **Shows permission prompt** - If undecided, displays a permission dialog
3. **Calls callback** - Invokes success or error callback based on user's choice

### Permission States

| State | Description | User Experience |
|-------|-------------|------------------|
| **Granted** | User previously allowed location | Location detected immediately |
| **Denied** | User previously blocked location | Error callback fires, alert shown |
| **Prompt** | First visit or reset permissions | Browser shows permission dialog |

<Warning>
  If the user denies location permission, the app shows a generic alert: "No se pudo obtener tu ubicación". The user cannot distinguish between denied permissions and actual geolocation errors.
</Warning>

## Error Handling

The current error handler is minimal from `useMyLocation.ts:14-16`:

```tsx src/hooks/useMyLocation.ts
() => {
  alert("No se pudo obtener tu ubicación");
}
```

### Error Types Not Differentiated

The Geolocation API can fail for multiple reasons:

- **Permission denied** - User blocked location access
- **Position unavailable** - GPS/network location unavailable
- **Timeout** - Location request exceeded 5-second timeout

The app doesn't distinguish between these errors, providing the same generic message for all cases.

<Accordion title="Improving Error Handling">
```tsx
navigator.geolocation.getCurrentPosition(
  (pos) => {
    const { latitude, longitude } = pos.coords;
    setPosition(new LatLng(latitude, longitude));
  },
  (error) => {
    switch(error.code) {
      case error.PERMISSION_DENIED:
        alert("Permiso de ubicación denegado. Actívalo en configuración.");
        break;
      case error.POSITION_UNAVAILABLE:
        alert("Ubicación no disponible. Verifica tu GPS y conexión.");
        break;
      case error.TIMEOUT:
        alert("Tiempo de espera agotado. Intenta nuevamente.");
        break;
      default:
        alert("Error desconocido al obtener ubicación.");
    }
  },
  { enableHighAccuracy: true, timeout: 5000, maximumAge: 0 }
);
```
</Accordion>

## Security Considerations

### HTTPS Requirement

The Geolocation API requires a secure context:

- **Works**: `https://` domains, `localhost`
- **Doesn't work**: `http://` (non-localhost)

<Warning>
  If you deploy the app over HTTP (not HTTPS), location detection will silently fail in modern browsers. Always use HTTPS in production.
</Warning>

### Privacy Best Practices

- **Request location only when needed** - The app requests location immediately on mount, even if the user never uses location-based features
- **Explain why** - Users are more likely to grant permission if they understand the benefit
- **Provide alternative** - Allow manual address entry as a fallback

## Browser Compatibility

The Geolocation API is widely supported:

- ✅ Chrome, Edge, Firefox, Safari (desktop and mobile)
- ✅ iOS Safari, Chrome Mobile, Samsung Internet
- ❌ Internet Explorer (requires polyfill)

<Info>
  All modern browsers support the Geolocation API, but accuracy varies based on device capabilities (GPS, Wi-Fi, cellular triangulation).
</Info>

## Performance Considerations

### High Accuracy Trade-offs

Setting `enableHighAccuracy: true` has implications:

- **Desktop**: Uses Wi-Fi positioning (if available) + IP geolocation
- **Mobile**: Activates GPS, which:
  - Provides better accuracy (within meters)
  - Takes longer to acquire (2-10 seconds)
  - Drains battery faster

### Timeout Configuration

The 5-second timeout is reasonable for most cases:

```tsx
timeout: 5000  // 5 seconds
```

- Too short (< 2s): May timeout before GPS acquires signal
- Too long (> 10s): Users wait too long, poor UX

## Limitations & Improvements

<Accordion title="No loading state">
  The app doesn't indicate when location is being fetched. Users on slow GPS may wonder if the app is working.
</Accordion>

<Accordion title="No retry mechanism">
  If location detection fails, there's no way to retry without refreshing the page.
</Accordion>

<Accordion title="Location requested immediately">
  Location is requested on app load, even if the user doesn't need it. This could be deferred until the user interacts with location-based features.
</Accordion>

<Accordion title="No continuous tracking">
  The app uses `getCurrentPosition()` (one-time) instead of `watchPosition()` (continuous). If the user moves, their marker doesn't update.
</Accordion>

<Accordion title="Generic error messages">
  All errors show the same alert. Users can't tell if they need to enable permissions or if there's a technical issue.
</Accordion>

## Advanced Usage

### Continuous Location Tracking

To track the user's location continuously as they move:

```tsx
useEffect(() => {
  const watchId = navigator.geolocation.watchPosition(
    (pos) => {
      const { latitude, longitude } = pos.coords;
      setPosition(new LatLng(latitude, longitude));
    },
    (error) => {
      console.error("Location error:", error);
    },
    { enableHighAccuracy: true, timeout: 5000, maximumAge: 0 }
  );

  return () => {
    navigator.geolocation.clearWatch(watchId);
  };
}, []);
```

### Adding a Loading State

```tsx
const [position, setPosition] = useState<LatLng | null>(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState<string | null>(null);

useEffect(() => {
  navigator.geolocation.getCurrentPosition(
    (pos) => {
      const { latitude, longitude } = pos.coords;
      setPosition(new LatLng(latitude, longitude));
      setLoading(false);
    },
    (err) => {
      setError(err.message);
      setLoading(false);
    },
    { enableHighAccuracy: true, timeout: 5000, maximumAge: 0 }
  );
}, []);

return { position, loading, error };
```

<Tip>
  Consider implementing a "Locate Me" button instead of automatic location detection. This gives users control and reduces unnecessary permission prompts.
</Tip>

## Testing Location Features

Modern browsers allow simulating locations:

**Chrome/Edge:**
1. Open DevTools (F12)
2. Press `Ctrl+Shift+P` / `Cmd+Shift+P`
3. Type "sensors"
4. Select "Show Sensors"

**Firefox:**
1. Open DevTools (F12)
2. Click the three dots menu
3. Settings → "Location" section
4. Enter custom coordinates
